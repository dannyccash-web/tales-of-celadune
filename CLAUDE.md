## Deployment (do this after every change set)

1. Commit and push to `main` at https://github.com/dannyccash-web/tales-of-celadune
2. GitHub Pages serves `main` branch root → https://dannyccash-web.github.io/tales-of-celadune/
3. Auth: Danny's GitHub token is stored in the local git remote URL (`git remote -v`) — not in any committed file. Never commit tokens.
4. Always verify the live URL after pushing (Pages can take ~1 min to update).
5. **Known quirk of this sandbox's mount of the project folder: it can't unlink (or rename) files** (confirmed: even a file just created with `touch` can't be `rm`'d or `mv`'d). Git relies on lock-file unlinks for `add`/`commit`/`push` (`.git/index.lock`, `.git/HEAD.lock`, `.git/refs/remotes/origin/*.lock`), so once any git command leaves one of these behind (e.g. a run that got interrupted), it's stuck forever and every future git write in the mounted `.git` fails with `Unable to create '.git/....lock': File exists`. `git status`/`log`/`diff` (read-only) keep working fine regardless — it's only writes that break, and a `GIT_INDEX_FILE` override alone isn't enough since HEAD/ref locks aren't affected by it. Reliable workaround: copy `.git` to a normal, writable filesystem (`/tmp`), strip any lock/tmp files from the copy, and point git at it with `--git-dir` while keeping `--work-tree` on the real project folder:
   ```
   WT="<absolute path to the project folder>"
   GD="/tmp/celadune_gitdir_$$_$(date +%s)"   # unique name — /tmp on this sandbox
                                                # has been observed to retain files
                                                # across sessions, don't reuse a name
   cp -r "$WT/.git" "$GD"
   find "$GD" -name "*.lock" -o -name "tmp_obj_*" | xargs -r rm -f
   git --git-dir="$GD" --work-tree="$WT" fetch origin
   git --git-dir="$GD" --work-tree="$WT" reset --soft origin/main   # IMPORTANT — see below
   git --git-dir="$GD" --work-tree="$WT" add -A
   git --git-dir="$GD" --work-tree="$WT" commit -m "..."
   git --git-dir="$GD" --work-tree="$WT" push origin main
   ```
   The `fetch` + `reset --soft origin/main` step matters: the mounted `.git`'s own `refs/heads/main` never gets updated by this whole workaround (only the copy's ref and the remote do), so on the *next* session the copy you make will start from a stale local `main` — a plain `commit` on top of it diverges from what's actually on `origin/main` and `push` gets rejected ("fetch first"). `reset --soft` rewinds the copy's branch pointer to the real remote tip while leaving the index/working tree (i.e. your actual file changes) untouched, so the following `commit` lands correctly on top of history that matches origin. Safe to run every time, even if nothing was stale.
   This leaves the mounted `.git` itself stale (still has its own stuck locks, now behind the new commit) — that's fine, it's only used read-only going forward from Claude sessions. If Danny works on this repo directly on his Mac, the stuck lock files are just normal Finder-deletable files there (this restriction is specific to how Cowork's sandbox mounts the folder), so `rm .git/*.lock` locally clears them.
   **Gotcha found 2026-07-08:** after `reset --soft origin/main`, `git status`/`ls-files` can show a batch of files that are actually fine on disk and in the target commit's tree as both staged-`D`eleted *and* untracked (`??`) at the same path — a stale/incomplete index carried over from a previous session's copy (confirmed via `git ls-files | wc -l` being smaller than `git ls-tree -r HEAD --name-only | wc -l`; the missing paths matched exactly). `reset --soft` alone doesn't fix this because if the source index was already short those paths, resetting only moves what's *there*. Fix: run `git read-tree HEAD` right after the reset (rewrites the index to exactly match HEAD's tree, touches nothing in the working directory) — then re-check `git status` before `add -A`/`commit`. Cheap enough to just always run after `reset --soft`, even if nothing was stale.

## Gameplay conventions

- Player speed 130 px/s; icon mirrors horizontally every 0.25s while moving (walk effect) — see WALK_FLIP_INTERVAL in world.js. Same flip applies to NPCs (`npc.moving`/`npc.walkTimer` set in `updateNpcs` by diffing position before/after each tick, since movement happens across several branches — fading/paused/waiting NPCs don't flip). Footsteps loop plays while walking (`audio.setWalking(bool)` each frame, player only).
- Door SFX plays when NPCs leave/return home AND when the player enters/exits an interior.
- Building labels: `buildings: [{label, x, y, r}]` in scene data — drawn on canvas when player within r, centered directly over the building's door (same x as the door, y offset toward the interior — flipped to *below* the door for north-facing doors like Old Barn's). Names on D3: Mirelle's Farmhouse, Tuckwell's House (was Hay Barn), Brenna's House (was Tool Shed), Storehouse, Well, Silo, Old Barn. No "Animal Pen" label — Gaffer lives there and the player just walks up to him instead.
- Soundtrack: js/audio.js, one looped track at a time, `audio.play(src, fadeMs)` cross-fades. Start screen plays celadune_theme.mp3 (retried on first gesture — browser autoplay), Start button cross-fades to celadune_overworld.mp3. Battle/scene tracks later use the same call. **Race condition fixed 2026-07-08:** both tracks could end up playing at once if the very first gesture was the same keypress that both unblocked the theme's autoplay retry *and* triggered Start — `play(theme)`'s success callback (a pending promise) could resolve *after* `play(overworld)` had already run synchronously in the same tick, and its unconditional `fade(el, musicVolume, fadeMs)` would re-fade the theme back in, clobbering the overworld transition's fade-out/pause on that same element (`fade()` cancels the other's `setInterval` via `el._fade`, so whichever fade call resolves later wins). Fixed by guarding the success callback the same way the existing failure `.catch()` already did: only fade in if `current.el === el` still holds, i.e. this track hasn't since been superseded by a newer `play()` call. Keep this guard if `play()` is ever refactored — it's the only thing preventing two tracks from both winning their fades.
- Collision maps are hand-traced from a 25px grid overlay on the scene background (PIL script: 25px pink lines, 50px cyan, 100px red, labels every 100px; inspect per-quadrant crops). CONSERVATIVE RULE (from Danny): if any part of an object touches a 25px cell, the whole cell is blocked — all rect edges are multiples of 25. The well rect is x325-400, y775-900 (frame + stone base only, corrected from an earlier trace that over-extended to y925 and ate into the path); the full path band (y900-985) below it is clear, no detour needed.
- Each of the 4 main crop fields (upper-left, upper-right, lower-left, lower-right) is split into 2 rects with a ~50px walkable dirt lane between them (row-to-row furrows are real but too narrow for the 36px collider even where clear — only wider worker-path-style gaps are worth opening). Lane positions were found by color-classifying plant vs. dirt pixels per column (green/red plant colors vs. tan/brown dirt) and picking the cleanest ~50px window, then eyeballed against a cropped zoom to confirm before committing — pure per-cell auto-detection was too noisy (different crops have different foliage density/color). Lanes: upper-left x275-325, upper-right x1425-1475, lower-left x250-300, lower-right's field west edge pulled in to x1350 (joins the pen/silo corridor below instead of a separate lane).
- Silo access: the pen (x1050-1250, y1000-1225 — see the retrace note below), the lower-right field (starting x1350 instead of x1300), and the grass south of the pen all leave the player multiple ways down to the silo: the ~55-100px corridor at x1250-1350 (main path down past the pen's east side), and now also the ground directly south of the pen (x1050-1175, y1225-1325, between the pen and the barn). That southern pocket is bounded on 3 sides (pen/silo/barn) and is exactly the spot CLAUDE.md previously warned not to open, because NPC steering can wander in and get livelocked (escaping needs a turn sharper than steer()'s ~126° limit) — confirmed again 2026-07-07 when opening it dropped Tuckwell's throughput from ~70 loops/10,000s to 8. It's now open for the player only: obstacles can carry an `npcOnly: true` flag (checked in `world.js`'s `blockersAt`) that's skipped for the player but still blocks every NPC, so the pocket is walkable for the player but invisible-walled for NPCs, none of whom ever legitimately path through it anyway. **Keep this pattern for any future "let the player somewhere NPCs shouldn't wander" fix** — it's safer than physically resizing obstacles NPCs rely on for containment.
- The pen itself is hollow, not solid: 4 thin (25px) fence rects around the x1050-1250, y1000-1225 footprint, with a clear interior at x1075-1225, y1025-1200 for Gaffer (the goat) to wander in. Keep any future pen edits hollow the same way — a solid pen rect would trap Gaffer's patrol targets inside an obstacle. **Retraced 2026-07-07:** the original trace (south rail at y1275-1300) made the collision box ~75px taller than the actual fence art (confirmed via a pixel-grid crop of D3_Background.jpg — the real fence sits at roughly y1000-1200), which let Gaffer visibly wander south of the real fence line before ever tripping a collision. Also updated Gaffer's 3rd patrol point from (1150,1240) to (1150,1160) to stay inside the corrected interior.
- Homes: NPC `home: {door, interior}` + `routine` steps (leaveHome/goto/wait/goHome) in scene data. `startsHome` (explicit bool on the NPC) decides whether they spawn inside — falls back to "routine[0] is leaveHome" if omitted, so old data still works. NPCs that spawn out in the world (Tuckwell, Brenna) set `startsHome: false` and their own `x/y`. Leaving/entering = door SFX + 0.7s fade (FADE_S). Spacebar at door: locked toast if NPC away, dialog over interior background if home (world.interior swaps Layer 1). NPCs at home don't render, collide, or take dialog focus.
- SFX are one-shots via `audio.sfx(src)` (no-op in node); soundtrack unchanged (one looped track, crossfade). Player must only walk paths/grass/dirt: crop fields, hedges, trees, buildings, barrels, well, pen, silo are all rects in js/data/<scene>.js.
- D3 NPCs: Mirelle (home/leaveHome loop: barn → well → home), Tuckwell (Tuckwell's House, formerly labeled Hay Barn; tours all 4 crop fields then home), Brenna (Brenna's House, formerly labeled Tool Shed; pen → silo → home), Old Gaffer (a goat — no home, just `patrol: [{x,y}...]` inside the pen's hollow interior; `world.js`'s existing back-and-forth patrol system, previously unused, covers this). All routine waypoints ride two shared clear corridors — the main path (y900-990 full width) and a couple of 50-75px north-south gaps between buildings (x400-450 by the well, x925-1000 by the barn lane) — never a straight line through a building.
- **Routine waypoint safety margin (important, learned the hard way):** every `goto`/door target needs ≥30px clearance from any obstacle edge, not just "technically not overlapping." COLLIDER is 36px (half=18), so a point only 18-20px clear is one stray avoidance deflection away from a multi-NPC livelock — two NPCs can wedge into a narrow gap and oscillate forever (each "moves" a little every tick, so the stuck-timer fallback never fires, and it never shows up in a short test). Keep shared-corridor waypoints near the center of the y900-990 band (~940) rather than hugging its edges. Validate new waypoints programmatically (check against `scene.obstacles` with the 36px collider) and stress-test with a long headless sim (thousands of simulated seconds, not just tens) before trusting a new route — the Tuckwell/Brenna livelock only appeared after ~400s of sim time.
- Collision test suites live in this workflow: probe walkable/blocked points with `world.blockersAt`, walk exits/lanes with simulated input, and run the 4 NPC steering scenarios (route-around, patrol, overlap-escape, wall-block) headless in node before every collision change. For multi-NPC routine changes, also run a long headless sim (`new World({getContext:()=>({})}, scene, {})`, drive `update()` in a loop for 1000s+ of sim time) checking for zero collision violations and no NPC stuck >3s outside an intended wait/fade/pause.
- Dialog animations: portrait slides in from the right + fades in via a CSS keyframe (`.portrait-enter` on `#dialog-portrait`, class removed + reflowed + re-added on every `openDialog` so it replays). NPC line types out character-by-character (`js/ui.js`, ~22ms/char); pressing Space/Enter while typing completes the line instantly instead of advancing, matching common RPG dialog UX.
- Menu & Inventory panels (`#menu`/`#inventory` in index.html, `.panel-box` styles in style.css, logic in `js/ui.js`): both share one layout — dark backdrop + centered 1500x850 double-gold-framed box (same frame recipe as `.dialog-box`, outer stroke bumped to 4px since these panels are much bigger than the dialog box and the original 2px read thin at that scale), a left vertical tab rail, and a right content area. Opening either pauses the world the same way dialog does (`ui.isAnyPanelOpen()` is OR'd into `main.js`'s `locked` flag); Escape or clicking the dark backdrop close it; opening one auto-closes the other. **No close button** — Danny wants everything playable with no mouse (controller-friendly), so the only ways to close are Escape or re-pressing I/M; see the keyboard-nav bullet below for the full scheme. Menu has Audio (Music/Effects sliders) and Stats (level, XP bar, attack/defense/speed/luck) tabs; Inventory has Equipment (headwear/gloves/armor/boots in a loose paper-doll grid — no character art exists to lay these over, so it's shape-only), Weapons (left/right hand), and Magic (6-slot grid) tabs. There's no equipment system yet, so every Equipment/Weapons/Magic slot renders as an empty gold-diamond `.slot-icon` placeholder (Items tab is a real system — see below). Sliders are custom (native `<input type=range>` made invisible and overlaid on the existing `.bar`/`.bar-fill` hex-clip component + a diamond `.slider-thumb` drawn from the value) rather than a styled native track, so they inherit the same look as the HUD bars for free.
- Audio volume is now live-adjustable: `js/audio.js` exposes `getMusicVolume`/`setMusicVolume`/`getSfxVolume`/`setSfxVolume` (0-1 master levels, defaults 0.75 music / 1.0 effects, matching the sliders' initial 75%/100% — updated 2026-07-07 from the original 80%/90%). `sfx(src, weight)`'s second arg changed meaning from an absolute volume to a relative weight multiplied by `sfxVolume` — if adding new one-shot SFX calls, pass a 0-1 weight, not a target volume.
- Per-character dialogue-open voice clips (2026-07-07): `audio.DIALOGUE_SFX` maps `npc.id` → an mp3 played once, at the moment their dialog opens (a quick audio flourish, not a full voice line). Wired centrally in `main.js`'s `openNpcDialog()`, which every dialog-opening call site (direct approach *and* home-door entry) now goes through, so the clip plays consistently regardless of how the NPC was reached. Add new NPCs' clips here rather than inlining `audio.sfx()` calls at each interact() branch.
- Dialog responses can carry an effect (2026-07-07): `dialog.responseEffects` is an optional array parallel to `dialog.responses` — `null`/omitted entries are inert (the existing placeholder "just closes" behavior), or an entry can be `{ damage: n, message: '...' }`. `ui.openDialog(npc, onClose, onResponse)` now takes a third callback, invoked with `(npc, selectedIndex)` right before the dialog closes; `main.js`'s `applyResponseEffect()` is the one place that interprets these (currently just Gaffer's "Pet Gaffer." response, which bites for 1 damage). Extend this same mechanism for future response effects (item gain, quest flags, etc.) rather than adding parallel logic.
- Health bar width now scales with `healthMax` instead of a fixed CSS size (2026-07-07, from Danny — "it will grow throughout the game as the player advances"): `ui.js`'s `HEALTH_PX_PER_POINT` (30px/point) sets `#health-bar`'s inline width from `stats.healthMax` on every `updateHud()` call. At the new 5-point starting max that's 150px (half the original fixed 300px); a future max-health increase (leveling, gear, etc.) just needs `stats.healthMax` bumped and the bar grows automatically. Magic bar still uses the fixed CSS width since it's hidden and unscaled for now — give it the same treatment if it's ever shown with a variable max.
- Damage flash (2026-07-07): `ui.flashHealthDamage()` (remove/reflow/re-add `.flash-damage` on `#health-fill`, same trick as the dialog portrait's entrance animation) triggers a brief brightness+glow pulse (`@keyframes health-flash` in style.css) layered on top of the fill's existing width transition, so the bar visibly glows while it animates down. Call it alongside `ui.updateHud()` anywhere player health decreases — currently only Gaffer's bite, but any future damage source (battle system, etc.) should do the same pairing.
- **Keyboard-only play (2026-07-07, from Danny — "the whole game should be playable without a mouse," reworked 2026-07-08 into an explicit level hierarchy per his spec):** `I`/`M` toggle Inventory/Menu from anywhere (also switch straight between them, or close, while one's open — see `ui.toggleInventory`/`toggleMenu`). Once a panel is open, navigation is a stack of levels, and **Escape always steps back exactly one level** rather than always closing the whole panel:
  - **Level 0 "tabs"** (the default on open, always the top tab): Up/Down cycle the active tab. Space enters its content (Level 1). Escape closes the whole panel.
  - **Level 1 "content"** (per-tab): Audio — Up/Down move between the Music/Effects rows, Left/Right adjust the focused one by `SLIDER_STEP` (5%). Items — Left/Right move tile focus, Space opens that tile's action popout (Level 2). Stats/Equipment/Weapons/Magic have no interactive content yet, so Level 1 there just waits for Escape. Escape returns to Level 0.
  - **Level 2 "popout"** (Items only): Up/Down move between the popout's enabled actions, Space confirms. Escape closes just the popout, back to Level 1 (the item grid) — unchanged from how this already worked.
  The gold arrow indicator moves with focus: `.panel-tab.active::before` shows only at Level 0 (hidden via a `.content-mode` class ui.js toggles on `.panel-tabs` when entering Level 1), and `.slider-row::before` (a matching gold triangle, new 2026-07-08) shows only on the focused row at Level 1 — so the arrow visibly "hands off" from the tab down into its content on Space, and back up on Escape. All of this lives in `ui.js`'s `panelKey()` + `setNavLevel()`, called from `main.js`'s keydown handler whenever `ui.isAnyPanelOpen()`. Mouse clicks bypass the level state where direct manipulation already works (tab clicks just select+reset to Level 0 since the sliders are draggable regardless of level; item tile clicks set Level 1 then open the popout in one step, since that's the only way to reach it with a mouse) — see the tab/tile click handlers in `initPanels`/`updateItemsPanel`. Any future interactive tab content should extend this same 3-level pattern rather than adding a parallel one.
- Hidden collectibles (2026-07-07): `scene.interactables` in scene data — invisible (no sprite) rects with a `label` and `reward`; same proximity-reveal pattern as building labels (`World.render()` draws the label within `range`, default `INTERACT_RANGE`=90), and `world.nearestInteractableInRange()` + `main.js`'s `interact()` grant the reward once on spacebar (marks `.collected` on the instance, not the scene data, so a fresh `World` always resets). First one: "A shiny object" in the grass north of Mirelle's Farmhouse, `{x:950,y:320,w:80,h:80}`, +3 gold.
- Magic HUD bar is hidden for now (`.stat.hidden` in index.html) — player hasn't learned magic yet. Markup is intact, just toggle the `hidden` class off to bring it back.
- Player starts at 5/10 health and 0 gold (`js/main.js`'s `stats` object) — a deliberate "new game" reset; level/XP/attack/defense/speed/luck are unrelated placeholder flavor values, untouched.
- **Panel chrome redesign (2026-07-07/08, matched to Danny's mockup):** dropped the per-tab-pane `<h2>` section headers from Menu/Inventory; the active tab is now marked by a small gold CSS-triangle arrow (`.panel-tab::before`, opacity toggled by `.active`) sitting to the left of the tab label, instead of the old left-accent bar. `.panel-content` padding tightened (70px/90px → 50px/60px) to use the room the header freed up.
- **Real item system (2026-07-07/08):** `js/data/items.js` is the game-wide item *catalog* (id → `{name, image, description, questItem}`), separate from inventory *state* (`js/main.js`'s `inventory` array, `[{id, qty}]`, mutated only via `addItem(id, qty)`/`removeItem(id, qty)`, both of which call `ui.updateItemsPanel(inventory, ITEMS)` to re-render). The Inventory panel's new "Items" tab (`ui.updateItemsPanel`) renders a 5-column tile grid (`.item-grid`/`.item-tile`/`.item-frame`/`.item-label`) from that state each time it changes — each tile shows the item's image, its *description* as the label (not just its name — matches Danny's spec), and a `(qty)` suffix when the player has more than one. Quest items (`catalog.questItem === true`) get a gold-corner-ribbon + black-star badge (`.quest-badge`, CSS border-triangle trick + a `\2605` `::after`).
- **Item action popout:** clicking (or Space-selecting, see below) a tile opens `#item-popout` (Use/Inspect/Remove/Cancel) positioned just under that tile — `ui.js`'s `positionPopout()` converts the tile's screen-space `getBoundingClientRect()` into 1920×1080 "stage space" by dividing by the current `#stage` CSS scale factor, since `#item-popout` lives inside `#ui` (which shares that scaled coordinate system). Remove is disabled (`.popout-action.disabled`, via a `data-quest` flag stashed on each tile) for quest items — ui.js reads that flag straight off the tile's dataset rather than importing the catalog itself, keeping ui.js presentation-only. `main.js`'s `onItemAction(itemId, action)` (registered via `ui.initItemsPanel({onAction})`) is the one place that interprets these — Inspect toasts the description, Remove calls `removeItem`, Use is a stub ("Nothing happens... yet.") until real consumables exist. Cancel and outside-clicks just close the popout without calling the handler.
- **Items-tab keyboard nav** extends the existing Menu/Inventory scheme (`ui.panelKey()`) rather than replacing it: on the Items tab, Left/Right walk the tile grid in a flattened row-major order (deliberately not true 2D nav, so it doesn't collide with Up/Down's existing tab-cycling role) and Space/Enter opens the popout for the focused tile. Switching tabs or closing a panel always closes any open popout too (`setActiveTab`/`closePanel` both call `closeItemPopout()`), so it never gets left open pointing at a stale tile.
- **"Received" item reveal (2026-07-08):** `ui.showReceivedItem(itemDef)` plays a gold-down-arrow + "RECEIVED" label + item-frame reveal (`#dialog-received`, `.received-enter` remove/reflow/re-add trick, `@keyframes received-drop`) anchored in the empty right side of the dialog response box, under the portrait — used the moment a `grantItem` response effect fires. `dialog.responseEffects` entries gained a new shape alongside the existing `{damage}`: `{ grantItem: 'item_id', qty?: 1, thankYou?: '...' }`. `main.js`'s `applyResponseEffect()` handles it by calling `addItem`, playing the reveal, and — if `thankYou` is present — calling `ui.updateDialogContent({line, responses})` to swap in a follow-up line/response set *without* closing the dialog box (portrait stays put); it signals this by **returning `true`**, which `ui.chooseResponse()` now checks before auto-closing (previously it closed unconditionally after every response — this is the one behavior change to that function).
